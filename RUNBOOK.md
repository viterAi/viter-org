# viter-org — Runbook

**Last updated:** 2026-05-14
**Audience:** A new engineer who needs to be productive on this codebase in one day.

This runbook is the answer to Shaul's question *"could another engineer pick this up cold?"*. Read this top to bottom; you should be able to clone, install, run locally, and deploy after ~1 hour.

---

## What this is

`viter-org` is the platform substrate Mordechai built for Viter. It's a Next.js 16 + React 19 monorepo containing:

- **`apps/web/`** — the user-facing UI: chat, meetings, spaces, settings
- **`packages/runtime/`** — extractors, synthesizers, citation parser, LLM client
- **`packages/orchestrator/`** — Trigger.dev async jobs (claim-extract, embed-claim, link-claim, mail-poll, more)
- **`packages/spec/`, `packages/ui/`, `packages/ontology/`** — shared types + UI primitives + ontology layer
- **`adapters/whatsapp-gowa/`** — the working WhatsApp adapter (508 LOC)
- **`adapters/{browser,desktop,file-watcher,hil,http,mcp}/`** — scaffolded adapter types (most are package.json only)
- **`infra/supabase/`** — edge functions + migrations for the viter-org Supabase project (`dkccadwohifcqcdzhhnu`)
- **`scripts/`** — backfills, smoke tests, ingestion utilities

Deploy: `vita-roan.vercel.app` (Vercel project `vita-platform` on viter team `viter2`).

The deeper architecture lives in `library/REPORT.md` + `library/bucket-a-viter/`. Read those before you start changing things.

## The L0 → L1 → L2 → L3 pipeline

The platform's core abstraction. Read `library/bucket-a-viter/l-pipeline.md` for the full picture.

**TL;DR for a new engineer:**

- **L0 (`l0_artifacts` table + Storage buckets):** raw source data — audio files, WhatsApp message dumps, uploaded PDFs, git commits
- **L1 (`l1_events` table):** atomic extracted events — one transcribed message, one commit, one extracted claim. Each row has `actor_id` + `channel_id` + `facet` (e.g., `messages`, `transcription`, `claim_facet`, `commit_message`, `tool_calls`)
- **L2 (`l2_syntheses` table):** AI-generated summaries scoped by `(scope_kind, scope_key)` — e.g., one row per day, per meeting, per WhatsApp thread. Body is markdown.
- **L3 (markdown files in `~/viter-workspace/l3/`):** cross-stream surfaces — `_now.md`, `_decisions.md`, `_morning-brief.md`, `_quotes.md`, `_concepts/`, `_people/`. Regenerated by the `index-l3` skill on every Claude Code session close.

A typical data flow: WhatsApp voice note arrives → GOWA webhook → `whatsapp-webhook` edge function → row in `l0_artifacts` + row in `l1_events` (facet=`messages`, modality=`voice`) → Trigger.dev `transcribe-voice` job → another `l1_events` row (facet=`transcription`) → Trigger.dev `claim-extract` job → `l1_events` rows (facet=`claim_facet`) → Trigger.dev `synthesize-day` job → row in `l2_syntheses` citing the event_ids → `index-l3` skill (next SessionEnd) → updated markdown files in `l3/`.

## Common operations

### Adding a new WhatsApp group to track

1. Add the group's GOWA `chat.id` to the channels table:
   ```sql
   insert into public.channels (tenant_id, kind, identifier, display_name)
   values ('<viter-tenant-uuid>', 'whatsapp', 'wa-group-<id>', '<friendly name>');
   ```
2. GOWA webhook will start landing messages for the group automatically (if GOWA is configured to forward all groups). Verify by checking `l1_events` for new rows in that channel within a few minutes.

### Adding a new principal (person)

```sql
insert into public.principals (tenant_id, kind, display_name)
values ('<viter-tenant-uuid>', 'human', 'Jane Doe');
```

Then map their phone number to the principal via:

```sql
insert into public.principal_aliases (principal_id, alias_kind, alias_value)
values ('<jane-uuid>', 'phone', '972501234567');
```

Future WhatsApp messages from that number will resolve to Jane's principal.

### Running a backfill from a WhatsApp zip export

```bash
cd ~/viter-workspace/viter-org
pnpm tsx scripts/ingest-zip.ts ~/Downloads/WhatsApp\ Chat\ -\ \<name\>.zip
```

This reads the zip, extracts attachments + chat, ingests into `l0_artifacts` + `l1_events`. Idempotent — re-running won't double-insert.

### Re-synthesizing a day's L2 from scratch

```bash
cd ~/viter-workspace/viter-org
pnpm tsx scripts/synthesize.ts day 2026-05-13
```

Calls OpenRouter, writes to `l2_syntheses`. Existing row is overwritten.

### Querying L1 events as substrate

Use the `lookup-l1` skill — recipes documented at `.claude/skills/lookup-l1/SKILL.md`. SQL examples for "show me everything Person X said in date range Y" with the correct attribution gotchas (Jeffrey's WhatsApp messages have actor_id=NULL; resolve via `metadata->>'push_name'`).

## Common bugs + how to debug them

### "WhatsApp webhook returning 401"

The GOWA webhook calls the Supabase edge function `whatsapp-webhook` with basic auth. Check:
1. `GOWA_BASIC_AUTH` env var on Vercel matches what GOWA is configured with
2. The Supabase edge function deployment is current (`pnpm supabase functions deploy whatsapp-webhook`)

### "Trigger.dev job stuck in queued"

The Trigger.dev account is on Mordechai's personal email. Log in to <https://trigger.dev> with that account. Check the project for queued tasks. Common cause: API rate limit on OpenRouter or AssemblyAI.

### "Magic link login redirects to localhost"

The Supabase Auth → URL Configuration redirect list determines where the magic link sends users. Production should have `https://vita-roan.vercel.app/auth/callback`. If a developer pushed code with a wrong `NEXT_PUBLIC_SITE_URL`, magic links generated from prod will redirect wrong. Fix the env var, redeploy.

### "L3 files aren't updating"

The `index-l3` skill runs on every Claude Code SessionEnd hook (`.claude/hooks/rebuild-pipeline.sh`). If your hook isn't running:
1. Check `~/viter-workspace/.claude/settings.json` — `SessionEnd` hook should reference `rebuild-pipeline.sh`
2. Check `~/viter-workspace/.claude/hooks/rebuild.log` for recent runs
3. Run manually: `bash ~/viter-workspace/.claude/hooks/rebuild-pipeline.sh`
